# QHarnesscc 设置系统 设计文档

- 日期：2026-06-26
- 状态：待评审
- 关联：`docs/specs/2026-06-22-QHarnesscc框架与schema设计.md`、`说明.txt`

## 1. 背景与目标

当前 GUI（`MainWindow`）只用 mock 引擎测试架子：`EngineThread` 硬编码创建 `MockProvider`/`MockTwoStageProvider` + `MockBashTool`，`workDir` 取自 `QDir::currentPath()`，全项目无任何配置存取代码。已实现的 `ProviderOpenAI`/`ProviderClaude` 未接入 GUI。

本任务为 QHarnesscc 引入**设置系统**：用一份 JSON 配置管理「思考模式开关 / LLM 接口（多 profile）/ 工作目录 / 加载的工具」，提供 Qt 设置对话框编辑，并让 `EngineThread` **按配置装配真实 Provider/Engine/工具**（值注入，取代硬编码 mock）。

## 2. 范围

### 2.1 包含
- `Settings` 数据结构 + JSON 序列化（`core/schema`）。
- `SettingsStore`：JSON 文件加载/保存（`core/config`，纯 C++，路径由 app 注入）。
- `SettingsDialog`：单 `QTabWidget` 设置对话框（`app`）。
- `MainWindow`：持有 store/settings、加「设置」入口。
- `EngineThread`：构造改为收 `Settings` 值拷贝，按配置装配 Provider/Engine/工具；未配 profile 自动回退 mock。
- 测试按钮「测试 ReAct」「测试两阶段ReAct」保留，改为配置驱动。
- `core/tests/test_Settings.cpp`：序列化与 Store 的 TDD 用例。

### 2.2 非目标（YAGNI，留作后续）
- 真实工具实现（本阶段工具仅 `MockBashTool`，「工具」Tab 只搭勾选框架）。
- provider 层 LLM thinking（Claude thinking block / 第三方 reasoner）——「思考模式」本任务仅指引擎层 `enableThinking`。
- 配置热生效（改配置后即时作用于运行中的引擎）；本任务为「下次启动引擎生效」。
- apiKey 加密/混淆（明文存储）。
- 多 profile 之外的「方案切换历史」「导入/导出」等。

## 3. 关键决策汇总

| 维度 | 决策 | 理由 |
|---|---|---|
| 范围 | 配置 + 驱动引擎 | 设置存了即能用，避免「存而不用」的割裂 |
| 访问模式 | 纯数据 + app 层 Store + 值注入（**非单例**） | 契合项目既有 DI 风格；避免 UI/worker 跨线程读写竞争；易测试 |
| 思考模式 | 引擎层 `enableThinking`（两阶段引擎的慢思考 Phase1 开关；引擎种类由按钮 `EngineKind` 决定） | 项目已实现，零 provider 改动 |
| LLM 配置 | 多 profile | 灵活支持多套模型/服务商 |
| 工具加载 | 勾选启用/禁用 | 贴合「加载的工具」语义，框架就绪 |
| 对话框 | 单 `QTabWidget` | KISS，常见做法 |
| 数据位置 | `core/schema/Settings`（数据）；`core/config/SettingsStore`（IO） | 数据归 schema，IO 独立，保持 schema 纯数据定位 |
| apiKey | 明文 | 本地桌面工具，KISS；可逆混淆是伪安全 |
| 测试按钮 | 保留并配置驱动，未配 profile 回退 mock | 过渡平滑，「测试」永远可跑 |
| 存储路径 | `<exeDir>/config/setting.json` | 用户指定 exe 同级 config 目录 |
| 生效时机 | 下次启动引擎 | 值注入天然语义，KISS |

## 4. 数据模型（`core/schema/Settings.h` + `.cpp`）

放在 `core/schema`，遵循该目录「纯数据 + ADL 序列化」惯例（同 `Message.h`）。`class` + 首行 `public:`，数据成员公开（nlohmann 自由函数与对话框需直接访问）。

```cpp
#ifndef QH_SCHEMA_SETTINGS_H
#define QH_SCHEMA_SETTINGS_H
#include <string>
#include <vector>
#include <optional>
#include <nlohmann/json.hpp>
#include "qh_export.h"

namespace qh {
namespace schema {

enum class ProviderType { OpenAI, Claude };
// roleToString/roleFromString 风格：providerTypeToString / providerTypeFromString
// + NLOHMANN_JSON_SERIALIZE_ENUM（置于 qh::schema 内、枚举定义之后）

struct LlmProfile {
    std::string  name;          // 用户命名，profile 唯一标识
    ProviderType providerType;
    std::string  baseUrl;
    std::string  apiKey;
    std::string  model;
    std::optional<double> temperature;   // nullopt = 不下发，用 provider 默认
    std::optional<int>    maxTokens;
};

struct Settings {
    std::vector<LlmProfile>  profiles;
    std::string              activeProfileName;   // 按 name 引用激活项；空 = 未激活
    bool                     enableThinking = false;
    std::string              workDir;
    std::vector<std::string> enabledTools;        // 启用的工具名子集
};

} // namespace schema
} // namespace qh
#endif
```

### 4.1 序列化语义（ADL `to_json`/`from_json`，置于 `qh::schema`）
- `LlmProfile`：全字段直读直写；`temperature`/`maxTokens` 为 `optional`，`nullopt` 时 omitempty。
- `Settings`：`profiles`/`activeProfileName`/`enabledTools`/`workDir` 空值 omitempty（首次默认配置文件干净）；`enableThinking` 默认 false 也可 omitempty。
- `providerType` 用 `NLOHMANN_JSON_SERIALIZE_ENUM`（"OpenAI"/"Claude"）。
- `std::vector<LlmProfile>` 等容器由 ADL 自动获得序列化。

## 5. 持久化（`core/config/SettingsStore.h` + `.cpp`）

纯 C++（`std::ifstream/ofstream`），**不依赖 Qt**；文件路径由 app 层注入（app 用 `QCoreApplication::applicationDirPath()` 拼出 `<exeDir>/config/setting.json`）。

```cpp
#ifndef QH_CONFIG_SETTINGS_STORE_H
#define QH_CONFIG_SETTINGS_STORE_H
#include <string>
#include "schema/Settings.h"
#include "qh_export.h"

namespace qh {
namespace config {

class QH_API SettingsStore {
public:
    explicit SettingsStore(std::string filePath);
    schema::Settings load() const;          // 文件缺失 → 默认 Settings；JSON 损坏 → 默认 + 记 lastError
    bool save(const schema::Settings& s) const;  // 失败（不可写等）→ false + 记 lastError
    std::string lastError() const;
private:
    std::string _filePath;
    mutable std::string _lastError;   // load/save 失败时写入，供 UI 提示
};

} // namespace config
} // namespace qh
#endif
```

行为约定：
- `load`：文件不存在 → 返回默认构造的 `Settings`，`lastError` 清空（属正常首次运行）。文件存在但 JSON 解析失败 → 返回默认 `Settings`，`lastError` 记录解析错误。`config/` 目录不存在不在 core 处理，由 app 在 save 前 `QDir().mkpath()`。
- `save`：序列化 → 直接写 `_filePath`（KISS，不做原子临时文件替换）；失败置 `lastError`、返回 false。

## 6. app 层装配与生命周期

- `MainWindow` 新增成员：`SettingsStore _store`、`schema::Settings _settings`。
- 构造时：用 `QCoreApplication::applicationDirPath() + "/config/setting.json"` 构造 `_store`；`_settings = _store.load()`；若 `lastError()` 非空，`_postMessage` 提示「配置损坏，已用默认」。
- 入口：菜单栏新增「设置(S)」菜单 →「设置…」项（`QAction`），触发打开 `SettingsDialog`。
- 对话框：在 `_settings` 的**副本**上编辑；`确定` → 回写 `_settings` + `_store.save()`，失败弹 `QMessageBox` 提示且不丢内存值；`取消` 不落盘。
- 测试按钮：「测试 ReAct」/「测试两阶段ReAct」仍各自触发 `test(kind)`，内部以 `_settings` + `kind` 构造 `EngineThread`（配置驱动，未配 profile 回退 mock）。
- 生效：下次「测试」/运行引擎时按当前 `_settings` 装配（值注入）。

## 7. 设置对话框（`app/SettingsDialog.h` + `.cpp`）

`QDialog`，内含一个 `QTabWidget`，四个 Tab：

- **LLM 配置**：左侧 `QListWidget` 列 profile（当前激活项标注）；`新增`/`删除` 按钮；右侧表单编辑选中项（名称、类型下拉 `OpenAI/Claude`、BaseUrl、ApiKey【`QLineEdit::Password` 模态】、Model、Temperature、MaxTokens）；每项可「设为当前」（写 `activeProfileName`）。重名校验拒绝。
- **引擎**：`QCheckBox`「开启慢思考（两阶段，剥夺工具的纯文本规划）」↔ `enableThinking`。
- **工具**：`QListWidget`（多选）列可用工具，勾选写 `enabledTools`。本阶段仅 `MockBashTool`。
- **工作目录**：`QLineEdit` + `浏览…`（`QFileDialog::existingDirectory`）↔ `workDir`。

`getSettings()`/`setSettings()` 负责对话框与 `Settings` 的双向同步；`确定` 时 `accept()` 并由 `MainWindow` 取回。

## 8. 引擎对接（值注入——本设计核心）

`EngineThread` 构造改为接收 **`Settings` 值拷贝**：worker 线程只读自己的拷贝，与 UI 线程**零共享**，无需任何锁（呼应「不用单例」的决策）。

```cpp
// 新签名：settings 提供配置，kind 由测试按钮传入（决定引擎种类）
EngineThread(QPostMessage* pm, std::string prompt, schema::Settings settings,
            EngineKind kind, QObject* parent = nullptr);
```

构造内（在所属线程，run 之前）按 `settings` 装配子树：
1. **Provider**：按 `activeProfileName` 在 `profiles` 中查找；
   - 命中且 `baseUrl`/`apiKey`/`model` 非空 → 按 `providerType` `new ProviderOpenAI(apiKey,baseUrl,model)` 或 `ProviderClaude(...)`；并按 `temperature`/`maxTokens` 调 `setTemperature`/`setMaxTokens`。
   - 未命中或缺字段 → 回退 `MockProvider`（两阶段则 `MockTwoStageProvider`），`_postMessage` 告警「未配置有效 profile，回退 mock」。
2. **Engine**（种类由按钮传入的 `kind` 决定；`enableThinking` 仅作用于两阶段引擎的 Phase1）：`kind==TwoStageReAct` → `Engine2StageReAct(provider, tm, workDir, settings.enableThinking)`；`kind==ReAct` → `EngineReActLoop(provider, tm, workDir)`。
3. **workDir**：取 `settings.workDir`（替代 `QDir::currentPath()`）；空则回退 `currentPath`。
4. **工具**：EngineThread 持「全部可用工具」表（本阶段仅 `MockBashTool` 实例），按 `enabledTools` 名字筛选 `registerTool` 子集；`enabledTools` 为空则不注册任何工具（engine 无工具可用）。

> 装配逻辑（profile 解析、回退判定）抽成纯函数，便于 `core/tests` 单测。

## 9. 安全与错误处理

- apiKey 明文存 `setting.json`。
- 文件缺失/JSON 损坏 → 默认 `Settings` + UI 提示，不崩溃。
- `save` 失败 → 对话框提示，内存 `_settings` 不丢。
- profile 重名 → 对话框校验拒绝；删除激活项 → `activeProfileName` 自动清空并提示「已取消激活」。
- 无可用 profile/工具 → 引擎回退 mock 或无工具运行，均不崩溃。

## 10. 测试计划（`core/tests/test_Settings.cpp`，TDD）

仿现有用例（`QH_TEST`/`QH_CHECK_EQ`）。注册到 `core/CMakeLists.txt` 的 `qharness_core_tests` 源列表。

- `Settings` ↔ JSON 往返：多 profile、`optional` omitempty、`ProviderType` enum、空集合 omitempty。
- `SettingsStore`：`save→load` 往返一致；`load` 不存在文件 → 默认且 `lastError` 空；`load` 损坏 JSON → 默认且 `lastError` 非空；`save` 到只读路径 → false 且 `lastError` 非空。
- 装配纯函数：按 `activeProfileName` 解析命中/未命中→回退；`kind` 选引擎种类、`enableThinking` 透传两阶段；`enabledTools` 筛选工具子集。
- UI（对话框）走手动验证。

## 13. 命名与约定遵循

- 命名空间：`qh::schema`（Settings 数据）、`qh::config`（SettingsStore）。
- include guard：`QH_SCHEMA_SETTINGS_H`、`QH_CONFIG_SETTINGS_STORE_H`、`QH_APP_SETTINGSDIALOG_H`。
- 成员变量下划线前缀：`_filePath`、`_settings`、`_store`。
- 源码 UTF-8（`/utf-8`）。
- 导出：`SettingsStore` 整类 `QH_API`；`Settings`/`LlmProfile` 为 `struct`（`qh_export.h` 已集中抑制 C4251）。
