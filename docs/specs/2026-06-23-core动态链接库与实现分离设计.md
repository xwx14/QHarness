# core 动态链接库化与实现分离设计

> 日期：2026-06-23
> 范围：`core` 模块（`qharness_core`）
> 背景：用户要求——函数实现移到 `.cpp`、`core` 模块改为动态链接库（DLL）。

## 1. 背景与目标

**现状**：
- `core` 当前是 `STATIC` 库（`add_library(qharness_core STATIC ...)`）。
- 5 个派生类的**构造函数与方法体都内联在头文件**（含 `/* TODO */` 占位）：`provider/ProviderOpenAI`、`provider/ProviderClaude`、`engine/EngineReActLoop`、`robot/RobotFeishu`、`memory/MemoryFile`。
- 基类（`Provider`/`Engine`/`Robot`/`Memory`/`Tool`/`ToolRegistry`）为纯虚 header-only，虚析构 `virtual ~X() = default;` 内联。
- `schema/Message` 与 `tool/ToolManager` 已是 `.h`/`.cpp` 分离。
- core 使用方：`skeleton_compile_check`（实例化全部派生类）、schema 测试（用 `to_json`/`from_json`/`roleToString` 等）、ToolManager 测试；`app` 经 `MainWindow` 暂不直接用 core。

**目标**：
1. `core` 由 `STATIC` 改为 `SHARED`（产出 `qharness_core.dll` + 导入库 `qharness_core.lib`）。
2. 所有内联函数体（含派生骨架的 `/* TODO */` 占位）移到对应 `.cpp`，头文件只留声明。
3. 保持 core 不依赖 Qt、测试全绿（63 checks/0 failures）、app 冒烟通过。

## 2. 核心决策（已与用户确认）

| 决策点 | 选择 | 理由 |
|---|---|---|
| 导出宏 | 手写 `QH_API`（`QH_BUILDING_DLL` 切换 `dllexport`/`dllimport`） | 用户选择；显式可控 |
| 实现分离范围 | 全部内联体（含 TODO 占位）移 `.cpp` | 彻底声明/实现分离，未来实现直接在 `.cpp` 写 |
| 导出粒度 | 整类 `class QH_API X` | 一次导出全部成员+vtable+RTTI，比逐方法简洁 |
| 输出目录 | DLL 与全部 exe 统一到 `build/out/<config>` | SHARED 后 exe 必须与 DLL 同目录才能运行 |
| 测试 exe 路径 | 从 `build/core/Release` 改为 `build/out/Release` | 找到 `qharness_core.dll` |

## 3. 导出宏机制（新建 `core/qh_export.h`）

```cpp
#ifndef QH_EXPORT_H
#define QH_EXPORT_H

#ifdef _WIN32
  #ifdef QH_BUILDING_DLL
    #define QH_API __declspec(dllexport)
  #else
    #define QH_API __declspec(dllimport)
  #endif
#else
  #define QH_API
#endif

#endif // QH_EXPORT_H
```

- `core/CMakeLists.txt` 注入编译定义：`target_compile_definitions(qharness_core PRIVATE QH_BUILDING_DLL)`。编译 core 时 `QH_API = dllexport`；app/tests 消费时未定义该宏，`QH_API = dllimport`。
- `qh_export.h` 置于 `core/` 根（include 根），各公共头 `#include "qh_export.h"`。
- include guard：`QH_EXPORT_H`（遵循 `QH_<文件>_H` 约定，因位于 include 根，模块段省略）。

## 4. 导出范围（`QH_API` 标记清单）

**整类导出**（`class QH_API X`）：

| 模块 | 类 |
|---|---|
| schema | `ToolCall`、`ToolResult`、`ToolDefinition`、`Message` |
| tool | `Tool`、`ToolRegistry`、`ToolManager` |
| provider | `Provider`、`ProviderOpenAI`、`ProviderClaude` |
| engine | `Engine`、`EngineReActLoop` |
| robot | `Robot`、`RobotFeishu` |
| memory | `Memory`、`MemoryFile` |

**自由函数导出**（声明前加 `QH_API`，实现在 `Message.cpp`）：
- `qh::schema`：`to_json`/`from_json` 共 4 组（`ToolCall`/`ToolResult`/`ToolDefinition`/`Message`）、`roleToString`、`roleFromString`。
- `ToolCall::parseArguments()` 作为 `ToolCall` 成员，随整类导出。
- `NLOHMANN_JSON_SERIALIZE_ENUM(Role)` 宏在命名空间内生成的内联 `to_json`/`from_json`，为内联函数，无需 `QH_API`。

**基类虚析构**：`virtual ~X() = default;` 暂保留内联；若 MSVC 报 vtable/RTTI 导出警告（如 `warning C4275`/`C4251`），则为基类补 out-of-line 析构（`.cpp` 内 `X::~X() = default;`），作为实现期调整项。

## 5. 实现分离（新建 5 个派生 `.cpp`）

派生类的构造函数与方法体内联体（含 TODO 占位）移到新建 `.cpp`，头文件只保留声明（构造函数声明、`override` 方法声明），私有数据成员保留在头（类定义需要）。

| 派生类 | 新建 `.cpp` | 移出的函数 |
|---|---|---|
| `ProviderOpenAI` | `core/provider/ProviderOpenAI.cpp` | 构造函数、`generate()` |
| `ProviderClaude` | `core/provider/ProviderClaude.cpp` | 构造函数、`generate()` |
| `EngineReActLoop` | `core/engine/EngineReActLoop.cpp` | `run()` |
| `RobotFeishu` | `core/robot/RobotFeishu.cpp` | 构造函数、`send()` |
| `MemoryFile` | `core/memory/MemoryFile.cpp` | 构造函数、`load()`、`save()` |

`schema/Message` 与 `tool/ToolManager` 已分离，仅补 `QH_API` 标记，无需新建 `.cpp`。

## 6. CMakeLists 改动

### `core/CMakeLists.txt`
- `add_library(qharness_core STATIC ...)` → `add_library(qharness_core SHARED ...)`。
- 源列表新增：`qh_export.h` + 5 个派生 `.cpp`（对应 `.h` 已在列表）。
- `target_compile_definitions(qharness_core PRIVATE QH_BUILDING_DLL)`。
- 设输出目录（VS 多配置自动追加 `Release`/`Debug` 子目录）：
  ```cmake
  set_target_properties(qharness_core PROPERTIES
      RUNTIME_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/out"   # .dll
      LIBRARY_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/out"   # .dll（Unix）/ 别名
      ARCHIVE_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/out")  # .lib 导入库
  ```
- core 仍**不依赖 Qt**；`/utf-8` 不变。

### `app/CMakeLists.txt`
- `qharness_app` 链接 `qharness_core`（现为 SHARED）+ Qt 不变；exe 已输出 `build/out`；windeployqt 部署 Qt，core DLL 已在同目录，无需额外部署。

### `core/CMakeLists.txt`（测试段）
- `qharness_core_tests` 增加 `RUNTIME_OUTPUT_DIRECTORY = ${CMAKE_BINARY_DIR}/out`（SHARED 后测试 exe 必须与 core DLL 同目录，此前在 `build/core/Release` 会找不到 DLL）。
- 链接 `qharness_core`（自动随 SHARED）。

## 7. 输出目录与 DLL 部署

- `qharness_core.dll`、`qharness_core.lib`（导入库）、`qharness_app.exe`、`qharness_core_tests.exe` 均输出到 `build/out/<config>`。
- app 运行：windeployqt 部署 Qt DLL 到 `build/out/<config>`，core DLL 已在同目录 → 开箱即运。
- tests 运行：core DLL 与测试 exe 同目录 → 直接可跑（core 不依赖 Qt，无需 Qt DLL）。

## 8. CRT 与跨 DLL 边界

- core/app/tests 均用 MSVC 默认动态 CRT（`/MD` Release、`/MDd` Debug），CMake 默认三者一致，无需特殊设置。
- `std::string`/`std::vector`/`std::unordered_map` 跨 DLL 边界传参：同 CRT 配置下安全。
- `nlohmann::json` 为 header-only 模板，在各编译单元实例化，跨边界 OK。
- 项目不跨 DLL 边界抛 C++ 异常或用 `typeid` 跨边界，故基类 `= default` 内联析构的 RTTI 一致性风险可接受（见 §4 调整项）。

## 9. 测试与验证

- 构建 `qharness_core`（SHARED）→ 产出 `qharness_core.dll` + `qharness_core.lib`（`build/out/Release`）。
- 构建+运行 `qharness_core_tests`（`build/out/Release/qharness_core_tests.exe`）：**63 checks / 0 failures**，含 `skeleton_compile_check` 通过（派生类经 core DLL 实例化、链接）。
- 构建+冒烟 `qharness_app`（`build/out/Release/qharness_app.exe`，Qt + core DLL 同目录）。
- 同步更新项目 `CLAUDE.md` 中测试 exe 路径约定（`build/core/Release` → `build/out/Release`）。

## 10. 坑点

- **测试 exe 路径变更**：SHARED 后测试 exe 移到 `build/out/Release`，`CLAUDE.md`「构建/测试/运行」段需同步更新。
- **DLL 必须与 exe 同目录**：`qharness_core.dll` 只在 `build/out/<config>`；`build/core/`、`build/app/` 下不再有可用 exe。
- **Anaconda Qt 干扰** 依然存在：app 仍只能在 `build/out/<config>` 运行（windeployqt 已部署正确 Qt）。
- **CRT 必须一致**：若未来手动改某目标的 `/MT` 静态 CRT，会与 core DLL 的 `/MD` 冲突，导致链接/运行错误。
- **基类 vtable 警告**：`QH_API` 抽象基类若触发 MSVC `C4251`/`C4275`，按 §4 补 out-of-line 析构。
- **手写宏的维护**：新增公共类须记得加 `QH_API`，否则消费方链接不到（`LNK2019`）。

## 11. 非目标（YAGNI，本次不做）

- 不实现派生类的真实业务逻辑（ReAct 循环、HTTP Provider、飞书推送、文件记忆）——仅把 TODO 占位体挪到 `.cpp`。
- 不引入 `GenerateExportHeader`（用户选手写宏）。
- 不做跨编译器/跨平台 DLL（Linux `.so`）的完整抽象——`qh_export.h` 仅 `_WIN32` 分支，非 Windows 退化为空宏（预留，不实际构建）。
- 不做 DLL 显式加载（`LoadLibrary`）/插件热加载。
- 不改变 core 不依赖 Qt 的边界。
